# macOS pkgbuild Packaging Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Add a repeatable local build path that produces a macOS installer package for qctl with package identifier `io.qpoint.qctl`.

**Architecture:** Keep the existing Bun wrapper build as the source of truth, then stage its compiled `bin/qctl` into a deterministic package root. Use Apple's built-in `pkgbuild` tooling to create a component package, with optional installer scripts only where the current CLI install flow cannot safely map to package installation semantics.

**Tech Stack:** Bun, TypeScript, Make, macOS `pkgbuild`, `pkgutil`, `plutil`, shell scripts.

---

## Current Constraints and Findings

- The project already builds a standalone wrapper binary with `make build`, producing `bin/qctl`.
- `pkgbuild`, `productbuild`, `plutil`, and `pkgutil` are available at system paths on this macOS host.
- The package identifier should be `io.qpoint.qctl`.
- Existing runtime service code uses LaunchDaemon label/path `com.qpoint.qctl`; decide whether to align this to `io.qpoint.qctl` as part of packaging, or keep the current launchd label stable and use `io.qpoint.qctl` only as the installer package id.
- Existing `qctl install` is designed for an interactive user context:
  - it runs qcontrol host initialization through sudo,
  - appends a qctl socket sink to the invoking user's qcontrol `run.toml`,
  - installs `/Library/LaunchDaemons/com.qpoint.qctl.plist`.
- macOS package scripts run as root and do not naturally know which user's qcontrol config should be modified. Avoid blindly writing root's `~/.config/qcontrol/run.toml`.

## Proposed Installer Shape

- Add packaging source under `packaging/`:
  - `packaging/root/usr/local/bin/qctl` should be generated during packaging, not committed.
  - `packaging/scripts/postinstall` and `packaging/scripts/preinstall` should be added only if needed and should be minimal.
- Add package output under `dist/` and ignore it.
- Add a script, likely `scripts/build-pkg.sh`, that:
  1. runs `make build`,
  2. creates a clean staging directory under `build/pkgroot`,
  3. copies `bin/qctl` to `build/pkgroot/usr/local/bin/qctl`,
  4. sets owner/group/mode metadata (`root:wheel`, executable mode),
  5. derives version from `package.json`,
  6. runs `pkgbuild --identifier io.qpoint.qctl --version <version> --root build/pkgroot --install-location / dist/qctl-<version>.pkg`.
- Add a Make target such as `make pkg` that delegates to `scripts/build-pkg.sh`.

## Open Design Decision: Package Postinstall Behavior

Running as root is useful for the host-level parts of installation: installing the binary, initializing qcontrol trust material, writing a root-owned LaunchDaemon, creating `/Library/Logs/qctl`, and creating `/var/run/qctl`. The risky part is only the user-scoped qcontrol configuration, because package scripts run as root and root's home/config is not the operator user's config.

Start with **Option A**: package `postinstall` performs only root/system setup. It must not configure any user's qcontrol `run.toml`, even if a console user is available.

This keeps the first package increment deterministic and safe for interactive installs, MDM installs, remote installs, and multi-user machines:

1. **Root-owned setup in postinstall:** run host-level setup directly as root. This installs or refreshes the LaunchDaemon with `QCTL_EXECUTABLE=/usr/local/bin/qctl`, points it at the stable package socket `/var/run/qctl/qctl.sock`, and creates log/runtime directories. Do not run qcontrol trust/user initialization from this package hook until that flow is separated from user keychain state.
2. **No user config mutation in postinstall:** do not inspect `/dev/console`, do not impersonate a user, and do not write `~/.config/qcontrol/run.toml` from the package script in the first increment.
3. **Separate user command:** add or plan a dedicated idempotent user command such as `qctl init-user` for the per-user qcontrol sink setup. Users can run it explicitly after package install, and future iterations can decide whether the app should prompt, warn, or self-heal when user config is missing.
4. **Clear installer output/docs:** postinstall should complete root setup and print clear next steps, for example `Run /usr/local/bin/qctl init-user as each user who should send qcontrol events to qctl.`

This suggests adding an explicit system command, for example `qctl install-system`, and having package `postinstall` invoke only that command. The existing interactive `qctl install` can remain separate until we intentionally redesign it around `install-system` plus `init-user`.

## Task 2: Add initial package build script

**Objective:** Create a reproducible script that builds `dist/qctl-<version>.pkg` with identifier `io.qpoint.qctl`.

**Files:**
- Create: `scripts/build-pkg.sh`

**Implementation notes:**
- Use POSIX shell, consistent with `scripts/download-qcontrol.sh`.
- Derive version from `package.json` using Bun or Node. Prefer Bun because the project already requires Bun:
  ```sh
  VERSION=$(bun -e 'console.log(require("./package.json").version)')
  ```
- Clean and recreate `build/pkgroot` and `dist` each run.
- Copy `bin/qctl` to `build/pkgroot/usr/local/bin/qctl`.
- Use `install -m 755` locally while staging. Do not require sudo for the build.
- Run:
  ```sh
  pkgbuild \
    --identifier io.qpoint.qctl \
    --version "$VERSION" \
    --root build/pkgroot \
    --install-location / \
    "dist/qctl-$VERSION.pkg"
  ```

**Verification:**
```sh
scripts/build-pkg.sh
pkgutil --pkg-info-plist dist/qctl-0.1.0.pkg | plutil -p -
pkgutil --payload-files dist/qctl-0.1.0.pkg
```

Expected:
- pkgbuild exits 0.
- pkg id is `io.qpoint.qctl`.
- payload includes `usr/local/bin/qctl`.

## Task 6: Add package postinstall for system setup only

**Objective:** Let the `.pkg` complete root-owned host setup without touching user qcontrol config.

**Files likely involved:**
- Modify: `src/installation.ts`
- Modify: `src/main.ts`
- Create: `packaging/scripts/postinstall`
- Modify: `scripts/build-pkg.sh` to pass `--scripts packaging/scripts` once the script exists.

**Design requirements:**
- Add an explicit system-level command such as `qctl install-system`.
- `postinstall` should call `/usr/local/bin/qctl install-system` and print the next user step.
- `install-system` installs or refreshes the LaunchDaemon and root-owned log/runtime directories without running qcontrol trust/user initialization.
- `install-system` must not append qctl sinks to any user's `run.toml`.
- `postinstall` must not inspect `/dev/console` or impersonate a user in the first increment.
- Keep the existing interactive `qctl install` separate until it is deliberately refactored to compose `install-system` and `init-user`.

**Verification ideas:**
- Use `pkgutil --expand` to inspect that `postinstall` is included and invokes only `install-system`.
- Use temporary paths where possible for tests of plist generation and system install helpers.
- Run `sudo installer -pkg dist/qctl-0.1.0.pkg -target /` only after payload and script inspection pass.
- After install, verify `/usr/local/bin/qctl --version`, LaunchDaemon presence, and clear installer logs/instructions for `qctl init-user`.

## Non-goals for the First Increment

- Do not notarize or sign the package yet.
- Do not auto-start the daemon from postinstall until root/system setup behavior is separated and tested.
- Do not modify user qcontrol config from package `postinstall` in the first increment; user setup belongs in explicit `qctl init-user`.
- Do not change qcontrol runtime behavior or forwarder behavior.
